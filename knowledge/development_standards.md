# Development Standards

## GUI変更時の必須アクション
- 新しいページ（`src/pages/*.tsx`）を追加したら、必ず対応するテストも作成する。
- テストファイルは `src/pages/__tests__/<PageName>.test.tsx` に配置する。
- 実装後は `python3 scripts/check_test_coverage.py` を実行し、ページ追加漏れやテスト不足がないか確認する。

## ヘルプページ更新ルール

### ルール: UIページ変更時はヘルプも必ず更新する

以下のいずれかの変更を行った場合、`web/frontend/src/pages/HelpPage.tsx` も更新すること:

1. **新しいページを追加した**: 新ページの説明を「各画面の使い方」タブに追加する
2. **既存ページに新機能を追加した**: 対応するページの説明を更新する
3. **UIの文言・構成を変更した**: ヘルプの説明がズレないよう修正する
4. **新しい設定項目を追加した**: 設定ページの説明、必要に応じてトラブルシューティングに追加する

### チェック方法

pre-commit フックで自動チェックされる:
- `web/frontend/src/pages/*.tsx`（HelpPage.tsx以外）が staged の場合、HelpPage.tsx も staged に含まれているか確認
- 含まれない場合は警告メッセージが表示される（コミット自体は止めない）

手動チェック:
```bash
python3 scripts/check_help_coverage.py
```

### ヘルプページの更新箇所

HelpPage.tsx の構成:
- タブ1「概要」: システム全体の説明
- タブ2「各画面の使い方」: 全ページの詳細説明
- タブ3「設定・CLI・トラブルシューティング」: プロバイダー設定方法とCLIコマンド

## API変更時の必須アクション
- 新しいエンドポイントを追加したら、`tests/test_web_server.py` に対応するテストを追加する。
- 少なくとも次の3パターンを網羅する: 正常系、異常系（404/500）、バリデーションエラー。

## 自律改善サイクルの原則
- 問題を発見したら、提案を作成する前に既存のテストで再現できるか確認する。
- ユーザーに報告する前に、自分で修正できないかを必ず検討する。

## パフォーマンス原則
- 複数のAPIコールでは、`Promise.allSettled`（独立して失敗できる場合）と `Promise.all`（すべて成功が前提の場合）のどちらが適切かを判断する。
- 1つのAPI失敗だけで画面全体が落ちる実装を避ける。

## セキュリティ原則
- APIキーはマスクして表示する（先頭8文字 + `...` + 末尾4文字）。
- 設定ファイルは `~/.repocorp/` 以下に保存し、リポジトリには含めない。
